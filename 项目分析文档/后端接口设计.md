# Cloud-Vault 后端接口设计

## 一、设计目标

后端 API 基于 Cloudflare Workers + Hono + TypeScript 实现，数据层使用 Cloudflare D1、KV、R2。接口设计需要同时满足 Web 端、后续移动端和管理后台调用。

本设计覆盖：

1. 接口命名规范。
2. 请求、响应、分页、错误码规范。
3. JWT + KV 会话鉴权规范。
4. 系统管理员和账本成员权限规范。
5. 核心业务的数据处理逻辑。
6. MVP 阶段主要接口清单。

---

## 二、基础约定

### 2.1 API 前缀

```text
/api/v1
```

初始化接口单独保留：

```text
/api/init/:secret
```

原因：初始化接口发生在系统正式可用前，不参与普通版本化 API。

### 2.2 数据格式

| 类型 | 规范 |
|------|------|
| 请求体 | JSON |
| 响应体 | JSON |
| 字段命名 | lowerCamelCase |
| 路径命名 | kebab-case + 复数资源名 |
| ID 类型 | 字符串，建议 ULID/UUID |
| 时间 | UTC ISO 8601 字符串 |
| 日期键 | `YYYY-MM-DD` |
| 金额 | 整数，最小货币单位，例如 35.00 元传 `3500` |
| 布尔值 | JSON boolean |
| 空值 | 可选字段允许 `null`，不建议用空字符串表达空 |

示例：

```json
{
  "amount": 3500,
  "currency": "CNY",
  "occurredAt": "2026-05-11T08:30:00.000Z",
  "dateKey": "2026-05-11"
}
```

### 2.3 HTTP 方法语义

| 方法 | 用途 |
|------|------|
| `GET` | 查询资源，不产生副作用 |
| `POST` | 创建资源或执行非幂等动作 |
| `PATCH` | 局部更新资源 |
| `DELETE` | 删除资源；默认软删除 |

不使用 `PUT` 作为 MVP 默认更新方式，避免要求客户端提交完整资源。

## 三、接口命名规范

### 3.1 资源路径

使用复数资源名：

```text
GET    /api/v1/books
POST   /api/v1/books
GET    /api/v1/books/:bookId/accounts
POST   /api/v1/books/:bookId/transactions
PATCH  /api/v1/books/:bookId/transactions/:transactionId
DELETE /api/v1/books/:bookId/transactions/:transactionId
```

### 3.2 从属资源

归属于账本的数据必须放在 `books/:bookId` 下：

```text
/api/v1/books/:bookId/accounts
/api/v1/books/:bookId/categories
/api/v1/books/:bookId/tags
/api/v1/books/:bookId/transactions
/api/v1/books/:bookId/budgets
/api/v1/books/:bookId/reports/summary
```

这样能强制所有账本数据接口都走账本权限校验。

### 3.3 动作接口

动作接口只在无法用标准 CRUD 表达时使用：

```text
POST /api/v1/auth/logout
POST /api/v1/books/:bookId/transactions/:transactionId/restore
POST /api/v1/books/:bookId/accounts/recalculate-balances
POST /api/v1/books/:bookId/ai/parse-transaction
```

动作名使用动词短语，kebab-case。

### 3.4 查询参数命名

查询参数使用 lowerCamelCase：

```text
GET /api/v1/books/:bookId/transactions?dateFrom=2026-05-01&dateTo=2026-05-31&type=expense&pageSize=30
```

常用分页参数：

| 参数 | 说明 |
|------|------|
| `page` | 页码，从 1 开始，适合管理后台 |
| `pageSize` | 每页数量，默认 20，最大 100 |
| `cursor` | 游标，适合账单列表滚动加载 |
| `sort` | 排序字段，如 `occurredAt` |
| `order` | `asc` 或 `desc` |

账单列表优先使用 cursor，后台列表可以使用 page。

## 四、请求规范

### 4.1 通用请求头

| Header | 必需 | 说明 |
|--------|------|------|
| `Content-Type: application/json` | 写接口必需 | JSON 请求体 |
| `Authorization: Bearer <token>` | 私有接口必需 | JWT |
| `X-Request-Id` | 可选 | 客户端请求 ID，未传则服务端生成 |
| `Idempotency-Key` | 可选 | 创建账单等写接口防重复提交 |
| `Accept-Language` | 可选 | `zh-CN`、`en-US` |

### 4.2 请求体校验

后端使用 Zod 做严格校验：

- 必填字段缺失返回 `VALIDATION_ERROR`。
- 类型不匹配返回 `VALIDATION_ERROR`。
- 未声明字段默认拒绝，避免前端误传字段被静默忽略。
- 金额必须为整数且大于 0。
- 所有 ID 必须是字符串。
- 时间必须是合法 ISO 8601。

### 4.3 幂等规范

以下接口建议支持 `Idempotency-Key`：

- 创建账单。
- AI 解析后确认创建账单。
- 批量导入账单。
- 上传附件元数据确认。

KV Key：

```text
idempotency:{userId}:{method}:{pathHash}:{idempotencyKey}
```

TTL 建议 24 小时。命中时直接返回第一次请求的响应摘要，避免移动端重复点击或网络重试造成重复账单。

## 五、响应规范

### 5.1 成功响应

统一响应包裹：

```json
{
  "code": "OK",
  "message": "ok",
  "data": {},
  "requestId": "req_01HXYZ",
  "timestamp": "2026-05-11T08:30:00.000Z"
}
```

说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | string | 成功固定为 `OK` |
| `message` | string | 成功固定为 `ok` 或本地化简短提示 |
| `data` | any | 业务数据 |
| `requestId` | string | 请求追踪 ID |
| `timestamp` | string | 响应时间 |

### 5.2 列表响应

```json
{
  "code": "OK",
  "message": "ok",
  "data": {
    "items": [],
    "pageInfo": {
      "page": 1,
      "pageSize": 20,
      "total": 120,
      "hasMore": true,
      "nextCursor": "eyJvY2N1cnJlZEF0Ijoi..."
    }
  },
  "requestId": "req_01HXYZ",
  "timestamp": "2026-05-11T08:30:00.000Z"
}
```

`total` 不是所有接口都必须返回。账单滚动列表可以只返回 `hasMore` 和 `nextCursor`，降低 D1 聚合压力。

### 5.3 错误响应

```json
{
  "code": "VALIDATION_ERROR",
  "message": "参数校验失败",
  "details": [
    {
      "field": "amount",
      "message": "amount must be greater than 0"
    }
  ],
  "requestId": "req_01HXYZ",
  "timestamp": "2026-05-11T08:30:00.000Z"
}
```

### 5.4 HTTP 状态码

| 状态码 | 使用场景 |
|--------|----------|
| 200 | 查询、更新、删除成功 |
| 201 | 创建成功 |
| 400 | 参数错误、业务前置条件不满足 |
| 401 | 未登录或 Token 无效 |
| 403 | 无权限 |
| 404 | 资源不存在或不可见 |
| 409 | 冲突，如邮箱已注册、邀请码已用完 |
| 422 | 业务规则校验失败 |
| 429 | 请求过于频繁 |
| 500 | 服务端异常 |

### 5.5 业务错误码

| code | HTTP | 说明 |
|------|------|------|
| `OK` | 200/201 | 成功 |
| `VALIDATION_ERROR` | 400 | 参数校验失败 |
| `UNAUTHORIZED` | 401 | 未登录 |
| `TOKEN_EXPIRED` | 401 | Token 过期 |
| `FORBIDDEN` | 403 | 无权限 |
| `NOT_FOUND` | 404 | 资源不存在 |
| `EMAIL_REGISTERED` | 409 | 邮箱已注册 |
| `INVALID_INVITE_CODE` | 422 | 邀请码不存在或错误 |
| `INVITE_CODE_EXPIRED` | 422 | 邀请码过期 |
| `INVITE_CODE_USED_UP` | 422 | 邀请码次数已用完 |
| `INVITE_EMAIL_MISMATCH` | 422 | 邀请码限定邮箱不匹配 |
| `ACCOUNT_ARCHIVED` | 422 | 账户已归档，不能记账 |
| `CATEGORY_ARCHIVED` | 422 | 分类已归档，不能记账 |
| `INSUFFICIENT_PERMISSION` | 403 | 账本角色权限不足 |
| `RATE_LIMITED` | 429 | 请求过于频繁 |
| `INTERNAL_ERROR` | 500 | 未预期异常 |

## 六、鉴权规范

### 6.1 公开接口

以下接口不需要登录：

```text
GET  /api/init/:secret
POST /api/v1/auth/register
POST /api/v1/auth/login
GET  /api/v1/public/config
```

OAuth 回调接口也属于公开入口，但首次绑定账号时必须校验邀请码。

### 6.2 JWT 规范

登录成功后返回 JWT。JWT payload 建议：

```json
{
  "sub": "user_01HXYZ",
  "email": "admin@example.com",
  "systemRole": "admin",
  "jti": "sess_01HXYZ",
  "iat": 1778488200,
  "exp": 1779093000
}
```

KV 保存会话状态：

```text
session:{jti} => { "userId": "user_01HXYZ", "expiresAt": "2026-05-18T08:30:00.000Z" }
```

鉴权流程：

1. 读取 `Authorization`。
2. 校验 Bearer Token 格式。
3. 使用 `jwt_secret` 校验 JWT 签名和过期时间。
4. 从 payload 读取 `jti`。
5. 查询 KV `session:{jti}`，不存在则视为已退出或被撤销。
6. 查询 D1 用户状态，确认未注销、未禁用。
7. 将 `currentUser` 注入 Hono context。

### 6.3 系统管理员权限

`users.system_role = admin` 才可访问：

```text
/api/v1/admin/*
```

管理员能力：

- 创建、禁用、删除邀请码。
- 查看邀请码使用记录。
- 查看用户列表。
- 禁用用户。
- 修改系统设置。

管理员不默认拥有用户私人账本访问权。除非后续明确增加审计/代管能力，否则管理员不能直接查看普通用户账单明细。

### 6.4 账本权限

所有 `books/:bookId/*` 接口必须校验 `book_members`：

```sql
SELECT role
FROM book_members
WHERE book_id = ?
  AND user_id = ?
  AND status = 'active';
```

账本角色：

| 角色 | 查询 | 新增账单 | 编辑账单 | 管理账户/分类 | 管理成员 | 删除账本 |
|------|------|----------|----------|---------------|----------|----------|
| owner | 是 | 是 | 是 | 是 | 是 | 是 |
| admin | 是 | 是 | 是 | 是 | 是 | 否 |
| editor | 是 | 是 | 是 | 是 | 否 | 否 |
| viewer | 是 | 否 | 否 | 否 | 否 | 否 |

权限函数建议：

```text
requireAuth()
requireSystemAdmin()
requireBookRole(bookId, ["owner", "admin", "editor"])
requireBookReadable(bookId)
```

### 6.5 限流规范

KV 记录限流计数：

| 场景 | Key | 建议限制 |
|------|-----|----------|
| 登录 | `rate_limit:login:{ip}` | 10 次 / 10 分钟 |
| 注册 | `rate_limit:register:{ip}` | 5 次 / 10 分钟 |
| 邀请码尝试 | `rate_limit:invite:{ip}` | 5 次 / 10 分钟 |
| AI 解析 | `rate_limit:ai:{userId}` | 60 次 / 小时 |
| 管理后台写操作 | `rate_limit:admin:{userId}` | 120 次 / 小时 |

### 7.2 处理链路

```text
request
  -> requestId
  -> CORS
  -> body size limit
  -> rate limit
  -> auth middleware
  -> permission middleware
  -> zod validation
  -> service
  -> repository
  -> response wrapper
```

### 7.3 Service 与 Repository 边界

| 层 | 职责 |
|----|------|
| routes | 绑定 URL、读取参数、调用校验、返回响应 |
| service | 业务规则、事务编排、权限后的领域逻辑 |
| repository | D1 SQL/Drizzle 查询，不放业务决策 |
| middleware | 鉴权、权限、限流、异常处理 |

## 八、核心数据处理逻辑

### 8.1 系统初始化

接口：

```text
GET /api/init/:secret
```

处理逻辑：

1. 校验 `secret === env.jwt_secret`。
2. 执行 D1 migrations。
3. 初始化 `system_settings`：
   - `registration_mode = invite_only`
   - `default_currency = CNY`
   - `default_locale = zh-CN`
4. 读取 `env.admin_email` 和 `env.registration_invite_code`。
5. 规范化邀请码：trim + 大写或小写统一。
6. 使用 `invite_hash_secret` 计算 HMAC-SHA256。
7. 如果不存在初始管理员邀请码，则插入：
   - `inviteRole = admin`
   - `allowedEmail = admin_email`
   - `maxUses = 1`
   - `usedCount = 0`
8. 写入 KV `init_lock`，避免重复初始化造成混乱。

响应：

```json
{
  "code": "OK",
  "message": "ok",
  "data": {
    "initialized": true
  },
  "requestId": "req_01HXYZ",
  "timestamp": "2026-05-11T08:30:00.000Z"
}
```

### 8.2 注册

接口：

```text
POST /api/v1/auth/register
```

请求：

```json
{
  "email": "admin@example.com",
  "password": "password123",
  "inviteCode": "ABCD1234",
  "nickname": "Admin"
}
```

处理逻辑：

1. 校验邮箱格式、密码强度、邀请码非空。
2. 邮箱统一转小写。
3. 邀请码规范化后计算 HMAC 哈希。
4. 查询 `registration_invites`：
   - `status = active`
   - `usedCount < maxUses`
   - 未过期
   - `allowedEmail` 为空或等于注册邮箱
5. 检查邮箱是否已存在。
6. 哈希密码。
7. 在同一业务事务中：
   - 创建 `users`，写入 `systemRole = invite.inviteRole`。
   - 写入 `registrationInviteId`。
   - 邀请码 `usedCount + 1`。
   - 创建默认个人账本。
   - 写入 `book_members` owner。
   - 创建默认账户「现金」。
   - 创建预设收入/支出分类。
8. 返回用户基础信息，不自动返回密码相关字段。

响应：

```json
{
  "code": "OK",
  "message": "ok",
  "data": {
    "user": {
      "id": "user_01HXYZ",
      "email": "admin@example.com",
      "nickname": "Admin",
      "systemRole": "admin"
    },
    "defaultBookId": "book_01HXYZ"
  },
  "requestId": "req_01HXYZ",
  "timestamp": "2026-05-11T08:30:00.000Z"
}
```

### 8.3 登录

接口：

```text
POST /api/v1/auth/login
```

请求：

```json
{
  "email": "admin@example.com",
  "password": "password123"
}
```

处理逻辑：

1. 校验邮箱和密码非空。
2. 查询用户，排除 `deletedAt != null`。
3. 校验密码哈希。
4. 生成 `jti`。
5. 生成 JWT。
6. 写入 KV `session:{jti}`。
7. 更新 `lastLoginAt`。
8. 返回 Token 和当前用户。

响应：

```json
{
  "code": "OK",
  "message": "ok",
  "data": {
    "token": "jwt-token",
    "expiresIn": 604800,
    "user": {
      "id": "user_01HXYZ",
      "email": "admin@example.com",
      "nickname": "Admin",
      "systemRole": "admin"
    }
  },
  "requestId": "req_01HXYZ",
  "timestamp": "2026-05-11T08:30:00.000Z"
}
```

### 8.4 创建账单

接口：

```text
POST /api/v1/books/:bookId/transactions
```

请求：

```json
{
  "type": "expense",
  "accountId": "acc_01HXYZ",
  "categoryId": "cat_01HXYZ",
  "amount": 3500,
  "currency": "CNY",
  "occurredAt": "2026-05-11T08:30:00.000Z",
  "note": "午饭",
  "merchantName": "公司食堂",
  "tagIds": ["tag_01HXYZ"]
}
```

转账请求：

```json
{
  "type": "transfer",
  "accountId": "acc_cash",
  "transferAccountId": "acc_bank",
  "amount": 100000,
  "currency": "CNY",
  "occurredAt": "2026-05-11T08:30:00.000Z",
  "note": "现金存入银行卡"
}
```

处理逻辑：

1. 校验用户是账本 `owner/admin/editor`。
2. 校验账户属于当前账本且未归档。
3. `expense/income` 必须有 `categoryId`，且分类属于当前账本、类型匹配、未归档。
4. `transfer` 必须有 `transferAccountId`，不能等于 `accountId`，分类必须为空。
5. 校验标签都属于当前账本。
6. 根据用户时区从 `occurredAt` 生成 `dateKey`。
7. 如果 MVP 只支持单币种，`baseAmount = amount`，`baseCurrency = currency`。
8. 在同一业务事务中：
   - 插入 `transactions`。
   - 插入 `transaction_tags`。
   - 更新相关账户 `cachedBalance`。
9. 返回账单详情。

余额变更规则：

| type | 主账户变化 | 转入账户变化 |
|------|------------|--------------|
| expense | `-amount` | 无 |
| income | `+amount` | 无 |
| transfer | `-amount` | `+amount` |

### 8.5 更新账单

接口：

```text
PATCH /api/v1/books/:bookId/transactions/:transactionId
```

处理逻辑：

1. 校验账本写权限。
2. 查询旧账单，确认属于当前账本且未删除。
3. 校验新账户、新分类、新标签合法。
4. 计算旧账单对账户余额的影响，并反向冲销。
5. 计算新账单对账户余额的影响，并应用。
6. 更新 `transactions` 和 `transaction_tags`。
7. 返回新账单详情。

注意：编辑账单时不要直接用新金额覆盖余额差值，转账、账户变更、类型变更都需要先冲销旧影响再应用新影响。

### 8.6 删除账单

接口：

```text
DELETE /api/v1/books/:bookId/transactions/:transactionId
```

处理逻辑：

1. 校验账本写权限。
2. 查询账单，确认未删除。
3. 写入 `deletedAt`。
4. 反向冲销该账单对账户余额的影响。
5. 保留 `transaction_tags`，恢复时可继续使用。

恢复接口：

```text
POST /api/v1/books/:bookId/transactions/:transactionId/restore
```

恢复时清空 `deletedAt`，重新应用余额影响。

### 8.7 账户余额重算

接口：

```text
POST /api/v1/books/:bookId/accounts/recalculate-balances
```

处理逻辑：

1. 仅 `owner/admin` 可执行。
2. 按数据库设计中的余额重算 SQL 重新计算每个账户余额。
3. 更新 `accounts.cachedBalance`。
4. 返回修复数量和账户余额列表。

该接口用于异常修复，不作为日常展示逻辑。

### 8.8 AI 自然语言记账

接口：

```text
POST /api/v1/books/:bookId/ai/parse-transaction
```

请求：

```json
{
  "text": "午饭花了35",
  "occurredAt": "2026-05-11T08:30:00.000Z"
}
```

处理逻辑：

1. 校验账本可读权限。
2. 加载当前账本分类、账户和用户默认货币。
3. 调用 Workers AI 或 DeepSeek。
4. 将 AI 输出约束到 JSON Schema。
5. 匹配最可能的分类和账户。
6. 写入 `ai_parse_logs`。
7. 返回账单草稿，不直接创建账单。

响应：

```json
{
  "code": "OK",
  "message": "ok",
  "data": {
    "draft": {
      "type": "expense",
      "amount": 3500,
      "currency": "CNY",
      "categoryId": "cat_food",
      "accountId": "acc_cash",
      "note": "午饭",
      "occurredAt": "2026-05-11T08:30:00.000Z"
    },
    "confidence": 86,
    "parseLogId": "ailog_01HXYZ"
  },
  "requestId": "req_01HXYZ",
  "timestamp": "2026-05-11T08:30:00.000Z"
}
```

客户端确认后，仍调用创建账单接口。

## 十一、安全注意事项

1. 所有 SQL 必须使用参数化绑定。
2. 所有账本资源必须带 `bookId` 并校验 `book_members`。
3. 管理员接口不能返回密码哈希、邀请码哈希、JWT 密钥等敏感字段。
4. 邀请码明文只在创建成功时返回一次。
5. 注册、登录、AI 解析必须限流。
6. 账号注销、导出、批量删除等操作需要二次确认，后续可接入邮箱验证码或 TOTP。
7. R2 文件访问使用短期签名 URL，不公开桶。
8. 错误响应不要暴露 SQL、堆栈、第三方模型原始异常。
9. OAuth 首次绑定账号必须走邀请码校验。
10. CORS 只允许配置的前端域名，开发环境可放宽。
