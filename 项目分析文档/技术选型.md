# Cloud-Vault 技术选型

## 一、前端架构 Frontend

| 模块 | 技术 | 说明 |
|------|------|------|
| Web 端 | Vue3 + TypeScript + Vite | 管理后台和主 Web 应用 |
| 移动端 | React Native | 后续移动端 App |
| UI/样式 | Tailwind CSS + 自定义 SVG 图标集 | 主打高质感与流畅动画 |
| 状态管理 | Pinia | Web 端用户状态、账本状态、偏好设置 |
| 图表 | ECharts | 收支趋势、分类占比、预算对比等 |
| PWA | Vite PWA | 离线缓存和移动端安装体验 |

## 二、后端架构 Backend - Serverless

| 模块 | 技术 | 说明 |
|------|------|------|
| 核心框架 | Hono | 轻量级、专为 Edge 设计，TypeScript 友好 |
| 运行环境 | Cloudflare Workers | 全球边缘节点部署 |
| API 风格 | RESTful API | MVP 阶段简单直接，后续可补 OpenAPI 文档 |
| 参数校验 | Zod | 注册、登录、账单、预算等接口入参校验 |
| 密码哈希 | bcryptjs 或 Web Crypto PBKDF2/Argon2 方案 | Worker 环境优先选择兼容 Edge Runtime 的实现 |
| JWT | jose | 生成和验证登录 Token |

## 三、数据层 Database & Storage

| 模块 | 技术 | 说明 |
|------|------|------|
| 主数据库 | Cloudflare D1 | Serverless SQLite，处理用户、邀请码、账本、账单等结构化数据 |
| 缓存与会话 | Cloudflare KV | 存储 JWT 会话状态、限流计数、用户偏好缓存 |
| 媒体存储 | Cloudflare R2 | 预留用于发票、收据、头像、备份和导出文件 |
| ORM | Drizzle ORM | 与 D1/SQLite 适配较好，类型安全，迁移清晰 |
| 迁移 | Wrangler D1 migrations | 管理建表和后续版本升级 |

## 四、注册准入与管理员初始化

参考 `cloud-mail` 的部署模式，Cloud-Vault 也应在部署阶段配置初始管理员和初始化密钥。但 Cloud-Vault 的注册入口默认不开放给所有人，必须使用管理员分发的邀请码。

### 4.1 部署变量

开发环境可写在 `wrangler.toml` 的 `[vars]` 中，生产环境建议使用 GitHub Actions Secrets 或 Cloudflare Workers Secrets。

```toml
[vars]
admin_email = "admin@example.com"
registration_invite_code = "change-this-invite-code"
jwt_secret = "change-this-jwt-secret"
invite_hash_secret = "change-this-invite-hash-secret"
registration_mode = "invite_only"
```

变量说明：

| 变量 | 必需 | 说明 |
|------|------|------|
| `admin_email` | 是 | 初始系统管理员邮箱 |
| `registration_invite_code` | 是 | 初始管理员注册邀请码，只用于首次管理员注册 |
| `jwt_secret` | 是 | JWT 签名密钥，也用于初始化接口校验 |
| `invite_hash_secret` | 建议 | 邀请码 HMAC 哈希密钥；MVP 可临时复用 `jwt_secret` |
| `registration_mode` | 是 | 默认 `invite_only`，表示注册必须填写邀请码 |

### 4.2 初始化接口

保留类似 `cloud-mail` 的初始化入口：

```text
GET /api/init/:secret
```

初始化流程：

1. 校验 `:secret` 是否等于部署环境中的 `jwt_secret`。
2. 执行 D1 数据库迁移。
3. 写入系统默认设置，默认 `registration_mode = invite_only`。
4. 将 `registration_invite_code` 哈希后写入 `registration_invites`。
5. 将该邀请码限制为 `allowed_email = admin_email`、`max_uses = 1`、`invite_role = admin`。
6. 初始化完成后，部署者使用 `admin_email + registration_invite_code` 注册，账号自动成为系统管理员。

### 4.3 注册流程

```text
用户填写邮箱、密码、邀请码
        ↓
后端校验邮箱、密码强度、邀请码哈希
        ↓
检查邀请码状态、有效期、剩余次数、邮箱限制
        ↓
创建用户并记录 registration_invite_id
        ↓
减少邀请码可用次数
        ↓
初始化默认个人账本、账户、分类
```

规则：

- 登录阶段不需要邀请码。
- 邀请码只在注册或 OAuth 首次绑定账号时使用。
- 数据库只保存邀请码哈希，不保存明文；建议使用 HMAC-SHA256，密钥来自 Worker Secret。
- 管理员可在后台生成普通用户邀请码。
- 初始管理员邀请码用完后不可再次注册管理员。

## 五、AI 交互层

| 模块 | 技术 | 说明 |
|------|------|------|
| 推理接口 | Cloudflare Workers AI | 优先使用 Cloudflare 原生能力 |
| 备用模型 | DeepSeek API | 通过统一 API 网关调用自然语言解析 |
| 解析结果 | JSON Schema | 约束金额、分类、账户、日期、备注等字段 |
| 日志 | D1 + 定期清理 | AI 解析日志保留 30-90 天，避免长期保存隐私输入 |

## 六、部署方式

### 6.1 手动部署

1. 创建 D1、KV、R2 资源。
2. 在 Worker 中绑定 `DB`、`KV`、`R2`。
3. 配置 `admin_email`、`registration_invite_code`、`jwt_secret`、`invite_hash_secret`。
4. 部署 Worker 和前端静态资源。
5. 访问 `/api/init/:jwt_secret` 初始化数据库。
6. 使用管理员邮箱和初始化邀请码完成首次注册。

### 6.2 GitHub Actions 部署

建议参考 `cloud-mail/doc/github-action.md` 的方式，在仓库 Secrets 中配置：

| Secret 名称 | 必需 | 用途 |
|-------------|------|------|
| `CLOUDFLARE_API_TOKEN` | 是 | Cloudflare API 令牌 |
| `CLOUDFLARE_ACCOUNT_ID` | 是 | Cloudflare 账户 ID |
| `D1_DATABASE_ID` | 是 | D1 数据库 ID |
| `KV_NAMESPACE_ID` | 是 | KV 命名空间 ID |
| `R2_BUCKET_NAME` | 否 | R2 存储桶名称，MVP 可暂不启用 |
| `ADMIN_EMAIL` | 是 | 初始管理员邮箱 |
| `REGISTRATION_INVITE_CODE` | 是 | 初始管理员注册邀请码 |
| `JWT_SECRET` | 是 | JWT 和初始化接口密钥 |
| `INVITE_HASH_SECRET` | 建议 | 邀请码哈希密钥 |
| `INIT_URL` | 否 | 自动初始化地址，如 `https://example.com/api/init/<JWT_SECRET>` |
